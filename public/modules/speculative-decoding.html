Speculative Decoding: Draft-and-Verify

A draft model proposes k tokens; the target verifies in one pass — lossless speedup.

Start Here

Let's start with the thing you watch every time a model streams an answer: tokens appear one after another, never all at once. That ordering isn't a UI choice — it's forced by how the model generates. And once you see why it's forced, a surprising opportunity opens up.

Here's the puzzle this module cracks. Generation feels slow, and the obvious fix — throw more GPUs at it — doesn't help a single stream at all. Stranger still, while the model grinds out one token at a time, the expensive hardware is mostly sitting idle. There's spare capacity right there, unused, every single step. What if you could spend it to produce several tokens for the price of one — without changing a single word the model would have said?

That's exactly what speculative decoding does, and it sounds impossible, so we'll earn it slowly. First we'll see why one token per step is forced and why that leaves the GPU idle, then the split between proposing tokens cheaply and verifying them in parallel, then the little piece of accept/reject math that makes it provably lossless — and finally when it gives you nothing at all.

How It Works

Start with why decode is slow in the first place. Autoregressive generation is sequential by construction: token N cannot be computed until token N-1 exists, because N-1 is part of the input that produces N. So you run one full forward pass through the entire model — all the layers, all the weights — to produce exactly one token, then repeat.

The bottleneck is not arithmetic. On modern GPUs a single-stream decode step is memory-bandwidth-bound: the expensive part is streaming the model's weight matrices (and the KV cache) from HBM into the compute units. Producing one token touches roughly the same weights as producing a hundred would — the tensor cores are mostly idle, waiting on memory.

This is the root fact that makes speculative decoding possible: there is spare compute sitting there per step.

The insight: separate proposing tokens from verifying them. Keep a small, cheap DRAFT model (say a 1B model of the same family) alongside the big TARGET model (the 70B you actually want to sample from). Each round, let the draft model autoregressively generate k candidate tokens — cheaply, because it is tiny.

Now the k proposed tokens are already known strings of text. That lets the target model process all k positions in ONE forward pass: feed the current context plus the k drafted tokens as a single sequence, and the target produces its next-token distribution at every one of those k positions in parallel.

This is the same reason training and prefill are fast — when the tokens are already fixed, attention over a whole span is one batched matmul, not k sequential steps. Verification is parallel precisely because the tokens being verified are already committed.

Now the accept/reject rule, and why it is LOSSLESS — not an approximation. Walk left to right through the k drafted positions. At each position you have the draft's probability q(x) for the token it proposed and the target's probability p(x) for that same token. Accept the drafted token with probability min(1, p(x)/q(x)). If it is accepted, move to the next position.

On the FIRST rejection, you stop, throw away all remaining drafted tokens, and resample that one position from the corrected distribution — a renormalized (p − q)⁺ — which is exactly the target's distribution conditioned on 'not the rejected draft.' This is modified rejection sampling, and the math guarantees the emitted token at every position is drawn from exactly p, the target model's distribution.

The draft never gets to inject a token the target wouldn't have sampled: where the draft is over-confident (q > p) it gets down-weighted, and where it disagrees the correction resamples from the target. Net effect: the sequence of tokens you emit is distributionally identical to plain autoregressive sampling from the target — speculative decoding changes SPEED, not WHAT the model would have produced. (Contrast this with the sampling module's temperature/top-p, which deliberately change the output distribution — speculative decoding is the opposite: it holds the distribution fixed.)

One speculative round (k=4), and where the speedup comes from
Context: "def add(a, b):\n    return"

Draft model proposes k=4 tokens (4 cheap draft passes):
    [ a  +  b  \n ]

Target model verifies ALL 4 in ONE forward pass:
    pos1  q(a)=0.7  p(a)=0.9   accept  (p>q, keep)
    pos2  q(+)=0.6  p(+)=0.8   accept
    pos3  q(b)=0.8  p(b)=0.5   accept w.p. min(1, 0.5/0.8)=0.625 -> REJECT
    pos4  ...discarded (never verified)

  Accepted this round: 2 tokens (a, +)
  Position 3 resampled from corrected target dist (p - q)+  -> e.g. "c"
  Emitted: a  +  c        (all drawn from the TARGET's distribution)

Cost of the round: 1 target forward pass (+ 4 cheap draft passes)
Tokens produced:    3   (2 accepted + 1 resampled)

  Naive decode:  3 tokens = 3 target forward passes
  Speculative:   3 tokens = 1 target forward pass
  -> ~3x fewer target passes THIS round.

The target forward pass is the expensive, memory-bound step.
Every accepted draft token is one target pass you did NOT have to run.

The economics: the speedup is governed by the acceptance rate α — how often the draft agrees with the target. Each round costs one target forward pass regardless of how many tokens it verifies (that is the whole point — the target pass is nearly free per extra position, because it is memory-bound). So the win is the expected number of tokens emitted per target pass.

With per-token acceptance probability α and a proposal length of k, the expected accepted-run length before the first rejection is roughly (1 − α^(k+1)) / (1 − α), plus the one guaranteed resampled token. Concretely: propose k=4, and if α ≈ 0.75 you accept ~2.5 tokens on average, so you emit ~3.5 tokens per target pass instead of 1 — call it a 2.5–3× wall-clock speedup after draft overhead. If α ≈ 0.4, you accept well under one token per round on average, the draft passes are pure overhead, and you can end up NET NEUTRAL OR SLOWER than plain decoding.

So α is everything, and α depends on draft–target ALIGNMENT: the draft must predict what the target would predict. That is why the best drafts are small models distilled from (or in the same family as) the target — a random 1B model paired with a 70B produces low α and no speedup. There is also a k tradeoff: bigger k means more tokens per accepted round but more wasted draft work when an early rejection discards the tail.

When it helps vs. when it does not. Speculative decoding buys back the idle compute of a memory-bandwidth-bound step. That describes single-stream, latency-sensitive, LOW-BATCH decode — one user watching tokens stream, batch size 1. There, the GPU has spare compute and speculative decoding converts it into throughput: big win.

But in already-saturated, HIGH-THROUGHPUT batched serving, many independent requests are already packed into each forward pass — the GPU's compute is BUSY, the step is no longer memory-bound, and the 'free' parallelism speculative decoding relies on is already consumed by the batch. Adding speculation there mostly adds draft overhead and rejected-token waste for little gain, and can reduce total throughput even as it might still trim single-request latency. So the same technique helps enormously at batch-1 and barely at all under a compute-saturated batch — the regime is the diagnosis.

Variants relax the two-model requirement: self-speculative decoding drafts using the target's own early layers or skipped layers; Medusa adds extra decoding heads to the target so it proposes multiple future tokens itself; EAGLE drafts in feature space for higher α. All keep the same lossless verify-and-accept core — they only change how the k proposals are produced.

The interactive lets you turn the acceptance rate α and the proposal length k and watch tokens-per-target-pass rise, then collapse to net-neutral when the draft falls out of alignment. Then the closing scenario puts it to work on a 70B assistant that's fast enough for batch but too slow live — see if you can confirm the 2–3× claim, size it, and call where it does nothing before the reasoning does.

Key Points

  • Single-stream decode is memory-bandwidth-bound, not compute-bound. One forward pass streams all the weights from HBM to produce just one token — the tensor cores sit idle, leaving spare compute to exploit.
  • Draft proposes, target verifies in parallel. A tiny draft model autoregressively generates k tokens; because those tokens are already fixed, the target scores all k positions in ONE batched forward pass — same parallelism as prefill/training.
  • The accept/reject rule is provably lossless. Accept each drafted token with prob min(1, p/q); on first rejection resample from the corrected (p−q)⁺ distribution. Every emitted token comes from exactly the target's distribution — speed changes, output doesn't.
  • Speedup is set by the acceptance rate α = draft–target alignment. High α (well-aligned/distilled draft) → 2.5–3×; low α (α≈0.4) → draft passes become pure overhead, net-neutral or slower.
  • Biggest wins are low-batch, latency-sensitive decode. Under high-throughput batched serving the GPU is already compute-saturated, the free parallelism is consumed by the batch, and speculation barely helps.
  • Variants change only how proposals are made: self-speculative (early/skipped layers), Medusa (extra decoding heads), EAGLE (feature-space drafting) — all keep the same lossless verify-and-accept core.

In Production — Apply It

Now let's put all of that to work on a real one. Your inference team ships a 70B model that generates at 30 tokens/sec on a single stream — fast enough for batch jobs, too slow for the live coding assistant where users watch every token appear. Buying more GPUs doesn't help: one stream can't be split across cards, and the GPU sits mostly idle during decode. A senior engineer proposes speculative decoding to 'make it 2–3× faster without changing the answers.' Take a moment before reading on: given everything above, is that claim believable, what sizes the speedup, and where will it do nothing? Here's the reasoning, step by step. The claim holds because single-stream decode is memory-bandwidth-bound — the GPU streams the weights to make one token while compute sits idle, so a cheap draft can propose several tokens the big model verifies in one parallel pass, and the accept/reject rule keeps every emitted token drawn from exactly the target's distribution (lossless — speed changes, answers don't). The size of the win is set by the acceptance rate α (draft–target alignment): a well-aligned draft gives ~2.5–3×, a poorly aligned one collapses to net-neutral. And it does nothing where the GPU is already busy: under high-throughput batched serving the batch already consumes the free parallelism, so the live batch-1 assistant benefits a lot while the nightly compute-saturated batch job barely moves.

Key Insight

Autoregressive decode is sequential and memory-bandwidth-bound — one target forward pass per token with the GPU mostly idle. Speculative decoding has a cheap draft model propose k tokens that the target verifies in a single parallel pass, keeping accepted tokens and resampling on the first rejection; the accept/reject math makes it LOSSLESS — exactly the target's distribution, faster, not different. Speedup is set by the acceptance rate α (draft–target alignment); it wins big on low-batch single-stream latency and barely helps under already-saturated high-throughput batched serving.
